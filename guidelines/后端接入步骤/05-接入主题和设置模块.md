# 步骤 5：接入主题和设置模块

目标：先完成最简单、最稳定的 CRUD 模块，建立整条前后端通路。

## 涉及页面

- `src/app/pages/ThemesPage.tsx`
- `src/app/pages/SettingsPage.tsx`

## 当前代码现状

- [`F:\samlay\KB-konwlege\src\app\pages\ThemesPage.tsx`](F:\samlay\KB-konwlege\src\app\pages\ThemesPage.tsx) 直接调用 store 的同步 `addTheme`、`updateTheme`、`deleteTheme`
- 主题卡片展示字段已经基本够用，主要问题是数据来源仍是 Mock
- 设置页虽然未在本步骤中展开，但当前分析结论是设置只存在内存，刷新会丢失

## ThemesPage 需要具体改什么

- 页面加载时调用 store 的 `loadThemes()` 或 `loadInitialData()`
- 保存主题时等待异步 action 完成，再关闭弹窗
- 删除主题时增加失败提示，避免后端删除失败但前端已误删
- 新建和编辑共用同一表单逻辑可以保留，不需要重写 UI

## SettingsPage 需要具体改什么

- 首次进入页面时拉取远端设置
- 表单初始值来自远端，而不是内存默认值
- 点击保存时调用 `saveSettings(payload)`
- 保存成功后更新 store，同步保留当前页面展示值

## 这一阶段建议新增的用户反馈

- 页面级 `Spin` 或按钮级 `loading`
- 保存成功 `message.success`
- 保存失败 `message.error`
- 空主题列表时的 `Empty` 占位

## 为什么先做这一步

- 主题和设置没有流式、轮询、文件上传这些复杂交互
- 最适合先验证 API 基础设施是否正确
- 一旦这里跑通，文档页和聊天页可以直接复用同样的请求和错误处理模式

## 具体动作

- 主题列表改为后端加载
- 新建、编辑、删除主题改为真实接口
- 设置页面改为加载远端配置并保存
- 保存成功和失败都要有明确反馈

## 优先这样做的原因

- 复杂度低
- 容易验证 API 层和 store 设计是否合理
- 能尽早暴露鉴权、错误处理、字段映射问题

## 完成标准

- 新建主题后刷新仍存在
- 设置修改后刷新仍保留
- 主题删除后相关页面状态不会残留脏数据
