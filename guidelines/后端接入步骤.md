# 后端接入步骤

本文基于 [`F:\samlay\KB-konwlege\后端接入分析.txt`](F:\samlay\KB-konwlege\后端接入分析.txt) 拆分前端接入真实后端的实施步骤，目标是把当前纯前端 Mock 项目逐步改造成可连接 FastAPI + ChromaDB + LLM 的真实知识库问答系统。

说明：本文件现在作为总览入口，具体执行时优先查看 [`F:\samlay\KB-konwlege\guidelines\后端接入步骤\README.md`](F:\samlay\KB-konwlege\guidelines\后端接入步骤\README.md) 和子文档。

## 步骤 1：统一接口契约

目标：先固定前后端的数据结构，避免前端改造到一半再反复返工。

需要确认的接口：
- 主题：`GET /api/themes`、`POST /api/themes`、`PUT /api/themes/:id`、`DELETE /api/themes/:id`
- 文档：`GET /api/documents`、`POST /api/documents/upload`、`POST /api/documents/url`、`POST /api/documents/text`、`DELETE /api/documents/:id`、`GET /api/documents/:id/status`
- 对话：`GET /api/conversations`、`GET /api/conversations/:id`、`POST /api/conversations`、`DELETE /api/conversations/:id`
- 问答：`POST /api/chat`
- 设置：`GET /api/settings`、`PUT /api/settings`

前端需要同步明确的响应字段：
- `Theme`、`Document`、`Conversation`、`Message`、`Settings`
- 文档状态值固定为：`pending | indexing | done | failed`
- 问答返回要包含：回答正文、消息 ID、引用来源 `sources`

完成标准：
- 前后端字段命名统一
- `src/app/types/index.ts` 能和后端 DTO 一一对应
- 聊天流式返回格式已经确定

## 步骤 2：配置开发代理和环境变量

目标：让前端开发环境可以稳定访问本地后端。

需要改造：
- 在 [`F:\samlay\KB-konwlege\vite.config.ts`](F:\samlay\KB-konwlege\vite.config.ts) 中增加 `/api` 代理
- 增加环境变量，例如：
  - `VITE_API_BASE_URL`
  - `VITE_SSE_BASE_URL`

建议：
- 本地开发走 Vite 代理，避免跨域
- 生产环境直接读取环境变量，不依赖代理

完成标准：
- 浏览器访问前端时，请求 `/api/*` 能转发到后端
- 不需要在组件里写死 `http://localhost:8000`

## 步骤 3：建立统一 API 层

目标：把零散请求能力集中到 `src/app/services/api.ts`，为页面和 store 提供稳定调用入口。

建议拆分：
- `apiClient`：axios 实例
- `themeApi`
- `documentApi`
- `conversationApi`
- `settingsApi`
- `chatApi`

API 层要具备的能力：
- `baseURL`
- 请求拦截器
- 响应拦截器
- 统一错误处理
- 上传进度回调
- SSE 聊天封装

建议输出函数：
- `getThemes()`
- `createTheme(payload)`
- `uploadDocument(formData, onProgress)`
- `createConversation(payload)`
- `streamChat(payload, handlers)`
- `getSettings()`

完成标准：
- 页面和 store 不再直接拼接口地址
- 所有请求都经由统一 API 层发起

## 步骤 4：改造 Zustand 数据层

目标：把 `src/app/store/index.ts` 从内存 Mock 仓库改成真实异步状态层。

需要调整的方向：
- 删除 `mockThemes`、`mockDocuments`、`mockConversations`
- 所有增删改查动作改成异步
- 增加全局或模块级的 `loading`、`error`
- 初始化时从后端拉取主题、文档、对话、设置
- 必要时使用 `persist` 做轻量本地缓存

建议改造方式：
- `loadInitialData()`
- `createTheme() / updateTheme() / removeTheme()`
- `loadDocuments(themeId?)`
- `createConversation() / loadConversationDetail(id)`
- `saveSettings(payload)`

状态层原则：
- 先调 API，再更新本地状态
- 乐观更新只用于低风险操作
- 删除主题时同步清理当前选中状态

完成标准：
- 刷新页面后数据来自后端而不是硬编码
- 页面组件不再依赖 mock 数据存在

## 步骤 5：接入主题和设置模块

目标：先完成最简单、最稳定的 CRUD 模块，建立整条前后端通路。

涉及页面：
- `src/app/pages/ThemesPage.tsx`
- `src/app/pages/SettingsPage.tsx`

具体动作：
- 主题列表改为后端加载
- 新建、编辑、删除主题改为真实接口
- 设置页面改为加载远端配置并保存
- 保存成功和失败都要有明确反馈

优先这样做的原因：
- 复杂度低
- 容易验证 API 层和 store 设计是否合理
- 能尽早暴露鉴权、错误处理、字段映射问题

完成标准：
- 新建主题后刷新仍存在
- 设置修改后刷新仍保留

## 步骤 6：接入文档上传和索引状态

目标：让知识库管理页真正能上传文档、导入网页、添加文本，并观察索引过程。

涉及页面：
- `src/app/pages/DocumentsPage.tsx`

需要改造：
- 去掉 `beforeUpload={() => false}` 的纯前端拦截逻辑
- 文件上传使用 `FormData`
- 网页导入调用 `POST /api/documents/url`
- 文本导入调用 `POST /api/documents/text`
- 上传完成后启动状态轮询 `GET /api/documents/:id/status`

建议补充能力：
- 上传进度条
- 状态轮询间隔，例如 2 到 3 秒
- 状态失败时展示错误信息
- 文档删除后同步刷新列表和主题统计

完成标准：
- 上传文件后后端能收到真实文件
- 文档状态会从 `pending` 进入 `indexing` 再到 `done/failed`

## 步骤 7：接入真实对话和 SSE 问答

目标：替换 `ChatPage.tsx` 里的假回复逻辑，接入真实 RAG 问答。

涉及页面：
- `src/app/pages/ChatPage.tsx`

核心改造：
- 删除 `simulateStreamingResponse()`
- 首次提问前先创建对话，或由 `/api/chat` 自动创建并返回 `conversationId`
- 发送问题时携带：
  - `themeId`
  - `question`
  - `conversationId`
  - `settings`
- 使用 SSE 按增量更新 assistant 消息内容
- 最终把 `sources` 引用信息渲染到消息卡片中

建议的流式事件：
- `start`
- `delta`
- `sources`
- `done`
- `error`

完成标准：
- 用户发送问题后能看到逐步生成的回答
- 刷新后历史对话可以重新加载
- 回答能展示引用来源

## 步骤 8：补齐历史记录、错误处理和验收

目标：把系统从“能跑”提升到“可用”。

需要补齐：
- `HistoryPage.tsx` 接入真实对话历史
- 网络失败、超时、SSE 中断的统一提示
- 空状态、加载状态、重试入口
- 删除对话、删除文档后的列表刷新
- 基本联调验收清单

建议验收顺序：
1. 创建主题
2. 上传文档
3. 等待索引完成
4. 发起问答
5. 查看引用来源
6. 刷新页面恢复历史
7. 修改设置并再次验证

完成标准：
- 主流程从主题创建到问答闭环全部可用
- 异常场景至少具备可见提示和可恢复路径

## 推荐实施顺序

建议按下面顺序推进，不要一开始同时改所有页面：

1. 接口契约
2. 代理和环境变量
3. API 层
4. Zustand 异步化
5. 主题和设置
6. 文档上传和索引
7. 聊天和 SSE
8. 历史记录、异常处理、验收
